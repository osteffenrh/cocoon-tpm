## Overview

This is the Cocoon TPM project - a Rust workspace for developing a software TPM (Trusted Platform Module). The codebase is designed for constrained execution environments (particularly stack-constrained), with all crates except CLI utilities being `[no_std]`.

## Build Commands

### Basic Development
```bash
# Build the entire workspace
cargo build

# Run all tests
cargo test

# Run tests for a specific crate
cargo test -p cocoon-tpm-crypto
cargo test -p cocoon-tpm-storage

# Run clippy (CI enforces -D warnings, but CI config has continue-on-error: true)
cargo clippy -- -D warnings

# Format code (uses custom settings from rustfmt.toml: max_width=120, wrap_comments=true)
cargo fmt
```

### BoringSSL Backend
The crypto crate supports an optional BoringSSL backend via FFI:
```bash
# Build with BoringSSL backend (requires cmake and objcopy from binutils)
cargo build --features boringssl

# Note: BoringSSL is distributed as a git submodule under bssl-bare-sys/
# Initialize submodules first if needed:
git submodule update --init --recursive
```

### TPM2 Interface Generation
The `tpm2-interface` crate contains autogenerated code. To regenerate:
```bash
cd tpm2-interface
./gen-interface.sh  # Requires gen-tpm2-cmd-interface tool from https://github.com/nicstange/gen-tpm2-cmd-interface-rs
```

### Fedora Packaging
The `.distro/` directory contains a Makefile for packaging workspace crates as Fedora RPMs. The Makefile provides an end-to-end workflow from Rust source to Copr builds:

```bash
cd .distro

# Generate Fedora spec files for all crates
make all

# Generate spec file for a specific crate
make crypto

# Build source RPMs
make srpms                          # All SRPMs
make cocoon-tpm-crypto.srpm         # Specific SRPM

# Build binary RPMs locally using mock
make rpms                           # All RPMs
make cocoon-tpm-crypto.rpm          # Specific RPM

# Upload and build in Fedora Copr
make coprs                          # All packages
make cocoon-tpm-crypto.copr         # Specific package

# Display information
make info                           # Show all crates, versions, and targets
make list-files                     # Show file and target lists
make help                           # Show all available targets
```

**Key Features**:
- Dynamically extracts version numbers from `Cargo.toml` files (no hard-coding needed)
- Uses `rust2rpm` to generate spec files from .crate files
- Uses `mock` for local RPM builds with chain building for inter-crate dependencies
- Uses `copr-cli` to upload and build packages in Fedora Copr (@virtmaint-sig/sev-snp-coconut)
- Secondary expansion for clean, declarative Make rules without `define`/`eval` patterns
- Dependency tracking: `Cargo.toml` → `.crate` → `.spec` → `.src.rpm` → `.rpm`/`.copr`

**Configuration Variables**:
- `FEDORA_RELEASE`: Fedora release number (default: 42)
- `MOCK_CONFIG`: Mock configuration to use
- `COPR_PROJECT`: Copr project for uploads
- `MOCK_REPO`: Additional repository for dependencies

**Note**: The `bssl-bare-sys` and `bssl-bare-sys-target-integration` crates are currently excluded from packaging (commented out in `CRATES` list).

## Architecture

### Workspace Structure
The project is organized as a Cargo workspace with the following crates:

- **utils-common**: Core utilities used across the project (`[no_std]`)
- **utils-async**: Async-related utilities including custom allocators and sync primitives (`[no_std]`)
- **tpm2-interface**: Autogenerated TPM2 interface definitions, marshalling/unmarshalling code (`[no_std]`)
- **crypto**: Cryptographic primitives with pluggable backends (pure Rust or BoringSSL FFI) (`[no_std]`)
  - Backend selection via features: default is pure Rust, enable `boringssl` for BoringSSL
  - Pure Rust backend: Uses RustCrypto for symmetric crypto, custom implementations for RSA/ECC built on cmpa crate
  - Extensive feature flags for algorithms: SHA variants, AES/Camellia/SM4, cipher modes (CBC/CFB/CTR/ECB/OFB), RSA schemes, ECC curves
- **storage**: Secure persistent storage with CocoonFs filesystem implementation (`[no_std]`)
  - Defines `NvFs` trait as common filesystem interface
  - CocoonFs: Special-purpose filesystem for secure TEE storage with journaling support
  - Format specification: `storage/src/fs/cocoonfs/cocoonfs-format.md`
- **bssl-bare-sys**: BoringSSL FFI bindings (compiles BoringSSL during build, renames symbols to avoid collisions)
- **bssl-bare-sys-target-integration**: Integration layer for customizing BoringSSL in embedded environments
- **cocoonfs-cli**: CLI utility for accessing CocoonFs images (not `[no_std]`)

### Key Architectural Patterns

**Dependency Flow**: utils-common → utils-async → tpm2-interface → crypto → storage
- Storage depends on crypto with specific features (CBC, ECB, SHA-512 for KDFa)
- Crypto depends on tpm2-interface for algorithm definitions

**Backend Abstraction**: The crypto crate uses feature-gated backend selection:
- Pure Rust backend in `crypto/src/pure_rust/`
- BoringSSL FFI backend in `crypto/src/bssl_ffi/`

**Storage Layers**:
- `storage/src/chip/`: Low-level chunked I/O abstractions
- `storage/src/fs/`: Filesystem layer with CocoonFs implementation
- CocoonFs components: inode index, extent management, auth tree, encryption entities, journal

**TPM2 Interface**: Uses macros for ergonomic error handling:
- `tpm_rc!()` - Instantiate TpmRc
- `tpm_err_rc!()` - Instantiate TpmErr::Rc
- `tpm_err_internal!()` - Instantiate TpmErr::InternalErr (panics in debug builds)

## Commit Requirements

All commits **must** include:
1. **Signed-off-by trailer** matching author name and email
2. **Non-empty commit body** (beyond just trailers)

This is enforced by CI via `scripts/check-signed-off.sh`. Use `git commit -s` to add the sign-off automatically.

## Pull Request Best Practices

When creating PRs:
- **Separate concerns**: Split large automated refactors (e.g., syntax changes like `if let && let`) from functional changes or CI configuration updates
- **Keep PRs focused**: Smaller, focused PRs are easier to review and less likely to cause merge conflicts
- Consider splitting style changes into separate PRs to keep reviews focused on logic and configuration changes

## Development Notes

- **Rustfmt**: Custom settings in `rustfmt.toml` (edition 2021, max_width 120, wrap_comments true)
- **Submodules**: BoringSSL is a git submodule - initialize with `git submodule update --init --recursive`
- **Stack constraints**: Code is designed to minimize stack usage for embedded/TEE environments
- **Feature flags**: The crypto crate has extensive feature flags - check `crypto/Cargo.toml` for available algorithms and curves
- **Workspace patches**: The root `Cargo.toml` patches all workspace crates via `[patch.crates-io]` for local development
